# Milestone 1 — Atomic Storage Layer & Protocol Log

> **My learning journal.** A deep walk-through of everything implemented in M1, the
> exact input/output contract of every method, the *why* behind each design
> choice, and interview-style questions to test myself later.

---

## 0. The one-paragraph mental model

I am building a **Delta-Lake-style transactional storage engine**. The big idea
of a "lakehouse" is: keep cheap, immutable data files (Parquet) in object
storage, and put a **transaction log** on top of them so that a pile of dumb
files starts behaving like a database table — atomic commits, snapshot isolation,
time travel, schema. **The data files never change in place.** Every change is a
*new* file plus a *new* log entry that says "as of version N, these files are
live." M1 builds the two foundations everything else stands on:

1. A **storage abstraction** with an *atomic create* primitive (`put_if_absent`).
2. A **protocol log** (versioned JSON commit files) that records table state as a
   list of *actions*.

Plus the Parquet read/write helpers and the statistics extraction that future
query pruning will rely on.

## 1. Architecture: the layers and the dependency rule

```
┌───────────────────────────────────────────────────────────────┐
│  tests/  (round-trip proofs: storage, commit-zero, stats)      │
└───────────────────────────────────────────────────────────────┘
                │ uses
┌───────────────────────────────────────────────────────────────┐
│  protocol/   log.py  (write_commit / read_commit)              │
│              actions.py (dataclasses + to_json/from_json)      │
│              paths.py (commit_path naming convention)          │
└───────────────────────────────────────────────────────────────┘
                │ uses                         │ uses
┌──────────────────────────────┐  ┌───────────────────────────────┐
│  parquet/io.py               │  │                               │
│  write_parquet               │  │                               │
│  read_footer_stats           │  │                               │
└──────────────────────────────┘  │                               │
                │ uses             │                               │
┌───────────────────────────────────────────────────────────────┐
│  storage/   base.StorageBackend  (abstract seam)              │
│             local.LocalFSBackend (real)                       │
│             s3.S3Backend         (stub, for M3)               │
└───────────────────────────────────────────────────────────────┘
                │
        OS filesystem / (later) S3 / MinIO
```

**The dependency rule (the most important rule in the repo):** every byte of I/O
goes through `StorageBackend`. Higher layers MUST NOT call `open()`, `os`,
`boto3`, or `fsspec` directly. This is stated right in
[base.py](../src/lakehouse/storage/base.py). It means I can swap LocalFS for S3
in M3 without touching the protocol or parquet code at all. This is the **Strategy
pattern** — `StorageBackend` is the strategy interface, the backends are concrete
strategies.

## 2. The end-to-end workflow

### 2.1 Write path (committing version 0 — see [test_commit_zero.py](../tests/test_commit_zero.py))

```
 caller
   │  1. build a pyarrow.Table (the actual rows)
   ▼
 write_parquet(table, backend, "data/test.parquet")
   │     serializes table → parquet bytes → backend.put(...)
   │     returns byte size
   ▼
 read_footer_stats(backend, "data/test.parquet")
   │     reads footer only → {numRecords, minValues, maxValues, nullCount}
   ▼
 build the action list:
   [ Protocol(reader=1, writer=2),
     Metadata(id, schema_string, format, partition_columns),
     AddFile(path, size, mtime, data_change, stats=json(...), partition_values),
     CommitInfo(timestamp, operation, params) ]
   ▼
 write_commit(backend, version=0, actions)
   │     to_json each action → newline-joined JSONL → utf-8 bytes
   │     backend.put_if_absent("_delta_log/00...0.json", bytes)   ← ATOMIC
   ▼
 commit is durable. _delta_log/00000000000000000000.json exists.
```

The crucial part: the commit lands via **`put_if_absent`**, not `put`. If two
writers both try to create version 0, exactly one wins; the loser gets
`FileAlreadyExists`. That single fact is the seed of the whole concurrency story.

### 2.2 Read path (reconstructing the table)

```
 read_commit(backend, version=0)
   │     backend.get("_delta_log/00...0.json")
   │     decode utf-8 → split lines → from_json each → list of actions
   ▼
 find the AddFile action(s) in the list
   ▼
 backend.get(add_file.path)   ← follow the LOG, never list the data dir
   ▼
 pq.read_table(...) → the original rows
```

**Hard rule baked into the test:** you reconstruct the table by *following the
`add` actions in the log*, never by `backend.list("data/")`. Listing a directory
to find "the data" is the classic data-lake bug — it sees half-written files,
files from aborted transactions, and files that were logically deleted. The log
is the single source of truth about what is live.

## 3. Module-by-module deep dive (every method's I/O contract)

### 3.1 `storage/base.py` — the abstract seam

`StorageBackend(ABC)` — six abstract methods. This is a pure interface; it has no
behavior, only a contract.

| Method | Input | Output | Contract / notes |
|---|---|---|---|
| `get(path)` | `str` relative path | `bytes` | Read whole object. Raise `FileNotFoundError` if missing. |
| `put(path, data)` | `str`, `bytes` | `None` | **Unconditional** overwrite-or-create. NOT for log files. |
| `put_if_absent(path, data)` | `str`, `bytes` | `None` | **Atomic create.** Raise `FileAlreadyExists` if it exists. The *only* primitive the commit protocol may use. |
| `exists(path)` | `str` | `bool` | Diagnostics / VACUUM only. Using it before a write is a **TOCTOU** bug. |
| `list(prefix)` | `str` | `list[str]` | Enumerate keys under a prefix. For log discovery + VACUUM, never to find live data files. |
| `delete(path)` | `str` | `None` | Remove an object. |

**Why two write methods?** Because they encode two different *intents*. `put` is
"I own this name, stomp whatever is there" (used for data files, which have unique
UUID-ish names so collisions can't matter). `put_if_absent` is "I am claiming
this name and I need to *lose loudly* if someone beat me to it" (used for commit
files, where the name is a sequential version number that two writers can race
for).

### 3.2 `storage/local.py` — `LocalFSBackend`

Constructor: `LocalFSBackend(root: str)` — all paths are resolved relative to
`root`. Tests point it at a pytest `tmp_path` for isolation.

| Method | What it actually does | Returns |
|---|---|---|
| `get(path)` | `open(root/path, 'rb').read()`. Re-raises `FileNotFoundError` with a friendly message. | `bytes` |
| `put(path, data)` | `makedirs(parent)`; write to a **unique temp file**; `os.replace(tmp, final)`; on any error, delete the temp. | `None` |
| `put_if_absent(path, data)` | `makedirs(parent)`; write+`flush`+`fsync` temp; `os.link(tmp, final)`; `FileExistsError` → `FileAlreadyExists`; `finally` deletes the temp. | `None` |
| `exists(path)` | `os.path.exists(root/path)` | `bool` |
| `list(prefix)` | returns `[]` if prefix missing; else `os.walk` and return every file as a path **relative to root**. | `list[str]` |
| `delete(path)` | `os.remove`; missing → `FileNotFoundError`. | `None` |

**The two atomicity mechanisms — this is the heart of M1:**

- **`put` uses `os.replace(tmp, final)`** — POSIX `rename(2)` is atomic *within a
  filesystem*. A reader either sees the entire old file or the entire new file,
  never a half-written one. Writing to a temp file first means a crash mid-write
  leaves a stray `.tmp`, not a corrupt `final`.

- **`put_if_absent` uses `os.link(tmp, final)`** — hard-linking is the trick.
  `link(2)` *fails atomically with `EEXIST`* if the target name already exists.
  There is no check-then-act gap. Compare with the naive `if not exists: write`,
  which has a **TOCTOU** (time-of-check-to-time-of-use) race: two writers both
  pass the `if`, both write, one silently clobbers the other. `os.link` collapses
  the check and the create into one indivisible kernel operation. After a
  successful link, `tmp` and `final` are two names for the same inode; the
  `finally` removes `tmp`, leaving `final`.

- **`fsync` in `put_if_absent`** forces the bytes from the OS page cache onto
  physical disk *before* the link makes the commit visible. Durability:
  if the machine loses power one millisecond after the commit "succeeds," the data
  is really there.

> **Honest note to self / known rough edge:** `put` builds the temp name as
> `path + f"{uuid}.tmp"` with no separator dot, so it's `foo<uuid>.tmp` rather
> than `foo.<uuid>.tmp`. Harmless (still unique) but slightly ugly. Also `put`
> does not `fsync` (only `put_if_absent` does); fine for M1 since data files are
> immutable and re-creatable, but worth knowing.

### 3.3 `storage/s3.py` — `S3Backend`

Every method raises `NotImplementedError`. It exists now only to prove the
abstraction is real and to be filled in during M3 (against the MinIO container in
[docker-compose.yml](../docker-compose.yml)). The interesting future question is
*how do you get `put_if_absent` on S3?* — historically S3 had no atomic
conditional create, which is the entire reason Delta/Iceberg needed external
coordination; modern S3 now supports `If-None-Match: *` conditional puts.

### 3.4 `protocol/paths.py` — naming convention

```python
commit_path(version: int) -> str   # "_delta_log/00000000000000000000.json"
```

| Input | Output |
|---|---|
| `0` | `_delta_log/00000000000000000000.json` |
| `7` | `_delta_log/00000000000000000007.json` |

**Why 20-digit zero-padding?** So that **lexicographic sort == numeric sort**.
Object stores and `list` return keys in string order. With zero-padding,
`...0009` sorts before `...0010`, so listing `_delta_log/` already gives commits
in version order. Without padding, `"10"` would sort before `"9"` and history
discovery would be wrong. 20 digits comfortably exceeds any realistic version
count (it's what Delta Lake itself uses).

### 3.5 `protocol/actions.py` — the action vocabulary

Six **frozen dataclasses** (immutable value objects). Each represents one fact
that can appear in a commit. They are "dumb data holders" — no behavior.

| Action | Fields | Meaning |
|---|---|---|
| `Protocol` | `min_reader_version`, `min_writer_version` | Version gate: refuse readers/writers too old to understand the table. |
| `Format` | `provider="parquet"` | The data file format. |
| `Metadata` | `id`, `schema_string`, `format`, `partition_columns` | Table identity + schema. |
| `AddFile` | `path`, `size`, `modification_time`, `data_change`, `stats`, `partition_values` | A data file that is **live** as of this commit. |
| `RemoveFile` | `path`, `deletion_timestamp`, `data_change` | A **tombstone** — a file that *was* live, now logically deleted (kept for VACUUM). |
| `CommitInfo` | `timestamp`, `operation`, `operation_parameters` | Audit breadcrumb for `history()`. Not load-bearing for state. |

`Actions` is a union type of all six.

**Serialization — the wire format:** each action serializes to a single-key JSON
object whose key names the type, e.g. `{"add": {...}}`, `{"remove": {...}}`.

```python
to_json(action: Actions) -> str
```
| Input | Output |
|---|---|
| `AddFile(path="data/x.parquet", size=550, ...)` | `'{"add": {"path": "data/x.parquet", "size": 550, ...}}'` |
| an unknown type | raises `ActionNotSupported` |

It maps `type(action)` → key via `class_map`, calls `dataclasses.asdict`, wraps,
`json.dumps`. (`asdict` recurses, so the nested `Format` inside `Metadata`
flattens to a plain dict automatically.)

```python
from_json(json_str: str) -> Actions
```
| Input | Output |
|---|---|
| `'{"add": {...}}'` | `AddFile(...)` |
| `'{"metadata": {..., "format": {...}}}'` | `Metadata(...)` with nested `Format` reconstructed |
| `'{"two": .., "keys": ..}'` | raises `ValueError` (must be exactly one top-level key) |
| `'{"bogus": {...}}'` | raises `ActionNotSupported` |

Note the special case: `Metadata`'s `format` field comes back as a dict, so
`from_json` re-wraps it into a `Format(**...)` before constructing `Metadata`.
This is the one place the recursive flattening of `to_json` has to be manually
un-done on the way back in.

**Why frozen dataclasses?** Immutability = safety. A committed action is a
historical fact; nothing should mutate it after the fact. Frozen also makes them
hashable and gives a free `__eq__` (which the tests lean on).

### 3.6 `protocol/log.py` — the commit reader/writer

```python
write_commit(backend, version: int, actions: list[dict]) -> None
```
- Serializes each action with `to_json`, joins with `\n`, adds a trailing
  newline → **JSONL** (one JSON object per line), encodes utf-8.
- Lands it via `backend.put_if_absent(commit_path(version), bytes)`.
- **Output:** `None`, but the side effect is the durable, atomic commit.
- **Raises:** `FileAlreadyExists` (propagated from the backend) if that version
  already exists — i.e. a conflicting concurrent commit.

```python
read_commit(backend, version: int) -> list[Actions] | None
```
- `backend.get(commit_path(version))`, decode, `splitlines()`, `from_json` each.
- **Output:** the list of action objects, or `None` if the commit file doesn't
  exist (`FileNotFoundError` is swallowed and turned into `None`).

> **Honest note:** the type hint says `Optional[Dict]` but it actually returns a
> `list` of actions (or `None`). Worth fixing the annotation to
> `Optional[List[Actions]]`.

**Why JSONL and not one big JSON array?** Append-friendliness and
fault-tolerance: each line is independently parseable, you can stream-read, and a
truncated final line damages only itself. It's the same format Delta Lake uses.

### 3.7 `parquet/io.py` — data files & statistics

```python
write_parquet(table: pa.Table, backend, path: str) -> int
```
- Writes the Arrow table to an **in-memory buffer** (`pa.BufferOutputStream`)
  with `pq.write_table`, pulls the bytes, hands them to `backend.put(path, ...)`.
- **Output:** the byte size (used to fill `AddFile.size`).
- One row group is fine for M1. Crucially it does **not** compute stats by
  iterating rows — Parquet already computed them while writing.

```python
read_footer_stats(backend, path: str) -> dict
```
- Reads the file bytes, wraps in `pa.BufferReader`, opens `pq.ParquetFile`, and
  walks **only `metadata` / row-group / column statistics** — never a data page.
- Aggregates **across row groups**: `numRecords` = sum of rows; `minValues` = min
  of per-group mins; `maxValues` = max of per-group maxes; `nullCount` = sum of
  per-group null counts.
- **Output shape:**
  ```python
  {"numRecords": int,
   "minValues":  {col: val, ...},
   "maxValues":  {col: val, ...},
   "nullCount":  {col: int, ...}}
  ```

**The all-null contract (pinned by a test):** for a column whose row group has no
real values, `has_min_max` is `False`, so that column simply **does not appear**
in `minValues`/`maxValues`, while `nullCount` still reports the nulls. This avoids
inventing a bogus min/max bound. (See
[test_parquet_stats.py](../tests/test_parquet_stats.py).)

**Why footer-only matters (the whole point of stats):** these min/max/null
summaries are what lets a future query engine do **data skipping / predicate
pushdown** — "this file's `col1` max is 2, my query wants `col1 > 100`, skip the
whole file without opening it." Reading the footer is cheap; reading data pages is
not.

> **Honest note / future S3 concern:** `read_footer_stats` currently does
> `backend.get(path)`, which pulls the *entire file* into memory and only then
> parses the footer. On local disk that's fine. On S3 you'd want a **range read**
> of just the trailing footer bytes; that needs a richer backend method later.

### 3.8 `errors.py` — the exception hierarchy

All inherit `LakehouseError`: `FileAlreadyExists`, `ConcurrentWrite`,
`SchemaMismatch`, `VersionNotFound`, `ActionNotSupported`. Only the first and last
are used in M1; the rest are placeholders for the commit-conflict and schema work
in later milestones. A single base class means callers can `except LakehouseError`
to catch anything from this engine.

## 4. Invariants I must never violate (carry these forward)

1. **All I/O goes through `StorageBackend`.** No `open`/`os`/`boto3` above the
   storage layer.
2. **Commits land only via `put_if_absent`.** Never `put` a log file.
3. **The log is the source of truth.** Find live data by reading `add` actions,
   never by `list`-ing the data directory.
4. **Data files are immutable.** Change = new file + new commit; old files become
   `remove` tombstones, deleted only by a future VACUUM.
5. **Version files are zero-padded to 20 digits** so lexical order = version order.
6. **Stats come from the footer, never from scanning rows.**

## 6. Interview questions & model answers

**Q1. Why do you need a transaction log at all? Why not just list the files in a
directory?**
*Listing is not transactional.* A directory listing sees partially-written files,
orphans from crashed jobs, and files that are logically deleted but not yet
garbage-collected. There's no atomic "swap in 3 files and remove 2" with a raw
directory. The log gives you a single, ordered, atomic record of exactly which
files are live at each version — enabling atomic multi-file commits, snapshot
isolation, and time travel. The files are dumb; the log makes them a table.

**Q2. What exactly makes `put_if_absent` atomic in `LocalFSBackend`, and why not
just `if not exists(): put()`?**
It uses `os.link(tmp, final)`. The POSIX `link(2)` syscall atomically fails with
`EEXIST` if the target name already exists — the existence check and the creation
are one indivisible kernel operation. The `if not exists: put` version has a
**TOCTOU race**: two writers can both pass the check, both write, and one
silently clobbers the other. `os.link` makes "create only if absent" a single
atomic step, so concurrent commits to the same version produce exactly one winner
and a clean `FileAlreadyExists` for everyone else.

**Q3. Difference between `put` and `put_if_absent`? When is each used?**
`put` is an unconditional overwrite-or-create (atomic *replace* via
`os.replace`). It's used for **data files**, which get unique names so collisions
are impossible and overwriting is never a hazard. `put_if_absent` is a
conditional atomic *create* that refuses to overwrite. It's used for **commit log
files**, whose names are sequential version numbers that two writers can race for;
refusing-and-raising is how we detect the conflict.

**Q4. Why are commit files named with 20-digit zero-padding?**
So lexicographic ordering equals numeric ordering. Object stores list keys in
string order; `"9"` sorts after `"10"` but `"...0009"` sorts before `"...0010"`.
Zero-padding lets a plain `list("_delta_log/")` return commits already in version
order, which is essential for finding the latest version and replaying history.

**Q5. Walk me through committing version 0 end to end.**
Write the Parquet data file with `write_parquet` (returns its byte size).
Extract footer stats with `read_footer_stats`. Build the action list:
`Protocol`, `Metadata` (id + schema + format + partitions), `AddFile` (path,
size, stats, etc.), `CommitInfo` (audit). Call `write_commit(version=0, actions)`,
which serializes the actions to newline-delimited JSON and writes them via
`put_if_absent("_delta_log/00...0.json")`. If that succeeds, the commit is atomic
and durable; if the version already existed, it raises `FileAlreadyExists`.

**Q6. How do you read the table back, and why is `backend.list("data/")`
forbidden?**
`read_commit(version)` to get the actions, find the `AddFile`(s), and
`backend.get(add.path)` to load each live data file by following the log.
Listing the data directory is forbidden because it would surface half-written,
orphaned, or logically-deleted files that aren't part of the committed snapshot —
breaking isolation and correctness. The log, not the filesystem, defines what's
live.

**Q7. `read_footer_stats` aggregates across row groups. How, and why footer-only?**
Each row group carries its own column statistics in the Parquet footer. We sum
`num_rows` for `numRecords`, take the min of per-group mins and max of per-group
maxes for `minValues`/`maxValues`, and sum per-group `null_count` for
`nullCount`. Footer-only because the footer already holds precomputed stats —
reading it is cheap, while reading data pages is expensive. These stats power
**data skipping**: a query can prune an entire file using its min/max bounds
without opening its contents.

**Q8. What should min/max be for an all-null column, and why?**
The column should be *absent* from `minValues`/`maxValues` (Parquet reports
`has_min_max == False`), while `nullCount` still records the nulls. Inventing a
sentinel min/max would corrupt data-skipping decisions; "no bound" is the honest
answer, and skipping logic can treat a missing bound as "can't prune."

**Q9. Why frozen dataclasses for actions, and why the `{"add": {...}}` envelope?**
Frozen = immutable: a committed action is a historical fact and must not mutate;
immutability also gives free `__eq__`/hashing the tests rely on. The single-key
envelope is a self-describing tagged union — the key names the action type, so
the deserializer knows which dataclass to build, and the JSONL format stays
extensible as new action types are added.

**Q10. Why does `fsync` matter in `put_if_absent` but the abstraction still
matters more?**
`fsync` forces bytes from the OS page cache to physical disk before the link makes
the commit visible, so a power loss right after "success" can't lose a committed
transaction (durability — the D in ACID). But the deeper point is the
`StorageBackend` seam: durability/atomicity are implemented *per backend*, so the
protocol layer never knows or cares whether it's on local disk or S3. That's what
lets M3 add S3 without touching a line of the commit or parquet code.

**Q11. The system "detects" concurrent writes but doesn't "resolve" them yet.
What's the difference and what comes next?**
Detection = the losing writer gets `FileAlreadyExists` because its version number
was taken. Resolution = re-reading the now-latest commit, checking whether the
other writer's changes conflict with mine (e.g., touched the same files), and if
not, **rebasing** my commit onto the next version number and retrying. That
optimistic-concurrency retry loop, plus `ConcurrentWrite`, is later-milestone
work.

**Q12. Where does this design break on S3, and how is it mitigated?**
Two spots. (1) `read_footer_stats` reads the whole object then parses the footer —
on S3 you'd want a byte-range read of just the footer. (2) `put_if_absent` relies
on `os.link`'s atomic `EEXIST`; classic S3 had no atomic conditional create
(the historical reason Delta/Iceberg needed external coordination), though modern
S3 now supports conditional `If-None-Match: *` puts. The `S3Backend` stub plus the
MinIO compose file are staged exactly to tackle this in M3.
```
